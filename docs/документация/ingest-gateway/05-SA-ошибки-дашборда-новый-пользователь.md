# Системный анализ: Ошибки на дашборде для нового пользователя

**Дата:** 2026-03-16
**Автор:** SA (Claude)
**Статус:** Исправлено и задеплоено на test
**Приоритет:** High

---

## 1. Описание проблемы

Новый пользователь, только что залогинившийся в систему, видит на главной странице (DashboardPage) ошибки вместо графиков и таблиц.

**Ожидаемое поведение:** Пустые виджеты с сообщением «Нет данных за выбранный период» или нулевые значения в метриках.

**Фактическое поведение:** Красные блоки ошибок с сообщениями типа «Ошибка загрузки…» и кнопкой «Повторить».

---

## 2. Анализ потока данных DashboardPage

### 2.1. Какие API-вызовы делает дашборд при загрузке

DashboardPage (`web-dashboard/src/pages/DashboardPage.tsx`) рендерит 7 компонентов, которые параллельно выполняют **до 10 HTTP-запросов** к ingest-gateway:

| # | Компонент | Endpoint | Параметры |
|---|-----------|----------|-----------|
| 1 | MetricsRow | `GET /dashboard/metrics` | — |
| 2 | GroupTimelineChart (APP) | `GET /dashboard/group-timeline` | `group_type=APP, from, to` |
| 3 | GroupTimelineChart (SITE) | `GET /dashboard/group-timeline` | `group_type=SITE, from, to` |
| 4 | TopUngroupedTable (APP) | `GET /dashboard/top-ungrouped` | `item_type=APP, from, to` |
| 5 | TopUngroupedTable (APP) | `GET /catalogs/groups` | `group_type=APP` |
| 6 | TopUngroupedTable (SITE) | `GET /dashboard/top-ungrouped` | `item_type=SITE, from, to` |
| 7 | TopUngroupedTable (SITE) | `GET /catalogs/groups` | `group_type=SITE` |
| 8 | TopUsersUngroupedTable (APP) | `GET /dashboard/top-users-ungrouped` | `item_type=APP, from, to` |
| 9 | TopUsersUngroupedTable (SITE) | `GET /dashboard/top-users-ungrouped` | `item_type=SITE, from, to` |

### 2.2. Маршрутизация запросов

```
Frontend (ingestApiClient) → базовый URL: /screenrecorder/api/ingest/v1/ingest
  → Traefik: StripPrefix /screenrecorder
  → nginx: proxy_pass /api/ingest/ → ingest-gateway:8084/api/
  → Controller: @RequestMapping("/api/v1/ingest/dashboard")
```

### 2.3. Проверка прав в контроллерах

Все эндпоинты DashboardController и CatalogController проверяют permission `CATALOGS:READ`:

```java
// DashboardController.java:33
DevicePrincipal principal = getPrincipalWithPermission(httpRequest, "CATALOGS:READ");
```

При отсутствии permission — бросается `AccessDeniedException` → **403 Forbidden**.

## 3. Выявленные корневые причины

### Причина 1: Миграция V31 не развёрнута на test/prod (КРИТИЧЕСКАЯ)

**Файлы миграций существуют в коде:**
- `auth-service/src/main/resources/db/migration/V31__add_catalogs_permissions.sql`
- `ingest-gateway/src/main/resources/db/migration/V31__app_groups_and_aliases.sql`
- `ingest-gateway/src/main/resources/db/migration/V32__app_group_browser_flag.sql`

**Но в MEMORY.md миграция V31 имеет статус «planned»:**
> V31 | App groups, aliases (ingest-gw) + CATALOGS:READ/MANAGE permissions (auth) | planned

**Статус миграций на test:** V1–V30, V34 (V31–V33 пропущены)

**Последствия отсутствия V31:**

| Компонент | Что происходит | Ошибка |
|-----------|---------------|--------|
| **auth-service** | Permissions `CATALOGS:READ` и `CATALOGS:MANAGE` отсутствуют в таблице `permissions` | JWT пользователя **не содержит** этих permissions |
| **ingest-gateway** | Таблицы `app_groups`, `app_group_items`, `app_aliases` **не существуют** в БД | SQL-запросы к этим таблицам падают с PostgreSQL error → **500 Internal Server Error** |
| **DashboardController** | `getPrincipalWithPermission("CATALOGS:READ")` — проверка permissions | Permission отсутствует в JWT → **403 Forbidden** |

#### Сценарий ошибки (пошагово):

1. Пользователь логинится → auth-service генерирует JWT
2. JWT содержит `permissions: ["DASHBOARD:VIEW", "DEVICES:READ", ...]` — **НЕТ `CATALOGS:READ`** (permission не существует в БД)
3. Фронтенд проверяет `DASHBOARD:VIEW` → ✅ маршрут доступен, DashboardPage рендерится
4. MetricsRow вызывает `GET /dashboard/metrics`
5. DashboardController проверяет `CATALOGS:READ` в JWT → **НЕТ** → `AccessDeniedException` → **403**
6. Frontend ловит ошибку → показывает красный блок «Ошибка загрузки метрик»
7. То же самое повторяется для всех 10 запросов → **весь дашборд в красных ошибках**

#### Даже если бы permission существовала:
Запросы `/dashboard/group-timeline` вызывают `ensureGroupsExist()` → `SELECT FROM app_groups` → таблица не существует → **500 Internal Server Error**.

**Вывод:** Без V31 миграции дашборд **гарантированно неработоспособен** для всех пользователей, не только новых.

---

### Причина 2: Новый пользователь OAuth без permission CATALOGS:READ

Даже после применения V31, новый пользователь, зарегистрированный через OAuth (Yandex), может не получить `CATALOGS:READ`.

**V31 выдаёт CATALOGS:READ ролям:**
- `SUPER_ADMIN`, `OWNER`, `TENANT_ADMIN`, `MANAGER`, `SUPERVISOR`

**Роли, которые НЕ получают CATALOGS:READ:**
- `OPERATOR`
- `VIEWER`

Если новому пользователю назначена роль `OPERATOR` или `VIEWER` — дашборд покажет 403 на все виджеты, хотя маршрут `/` доступен (требует `DASHBOARD:VIEW`, которая есть у обеих ролей).

**Несоответствие:** Маршрут требует `DASHBOARD:VIEW`, а API требует `CATALOGS:READ`. Пользователь видит страницу, но все данные — с ошибками.

---

### Причина 3: SUPER_ADMIN без tenant_id (граничный случай)

`resolveEffectiveTenantId()` в DashboardController:

```java
if (principal.hasScope("global")) {
    return tenantIdParam != null ? tenantIdParam : principal.getTenantId();
}
```

Если `SUPER_ADMIN` залогинен в свой тенант `prg-platform` (системный) — запросы выполняются, но данных нет (тенант пустой). Это не ошибка, но может выглядеть как «ничего не работает».

---

### Причина 4: Пустые данные отображаются как ошибка (UX-дефект)

Если V31 применена и permissions корректны, бэкенд возвращает **200 с пустыми данными** для нового тенанта:

```json
// /dashboard/metrics
{"connected_devices":0, "total_devices":0, "active_users":0, "tokens_used":0, "tokens_total":0, "video_size_bytes":0}

// /dashboard/group-timeline
{"groups":[], "days":[]}

// /dashboard/top-ungrouped
{"items":[], "total_count":0}
```

Фронтенд-компоненты корректно обрабатывают пустые данные — показывают «Нет данных». **Этот сценарий НЕ является причиной ошибок.**

| Причина | Затронутые пользователи | Ошибка | Код ответа | Критичность |
|---------|------------------------|--------|-----------|-------------|

## 5. Результат расследования на live-среде

Проверка на test-стейджинге показала:
- Миграции V31+V32 **уже были применены** ранее (permissions существуют, таблицы созданы)
- Все пользователи (OWNER, TENANT_ADMIN) имеют `CATALOGS:READ` в JWT
- Все dashboard-эндпоинты возвращают **200 OK** с данными или пустыми коллекциями
- Web-dashboard содержит актуальный JS-бандл со всеми dashboard API вызовами

**Первоначальная гипотеза о V31 не подтвердилась** — миграция уже была применена.

**Подтверждённая проблема:** DashboardController проверял `CATALOGS:READ` вместо `DASHBOARD:VIEW`. Для ролей OPERATOR и VIEWER (у которых есть `DASHBOARD:VIEW`, но нет `CATALOGS:READ`) все виджеты возвращали 403.

**Выполненный фикс:**
- Изменён `DashboardController.java`: `CATALOGS:READ` → `DASHBOARD:VIEW`
- ingest-gateway пересобран и задеплоен на test (2026-03-16)
- Все 5 dashboard-эндпоинтов возвращают 200 OK после фикса

### 5.2. Рекомендации по улучшению (после фикса)

#### R-3: Согласовать permission маршрута и API

**Проблема:** Маршрут `/` требует `DASHBOARD:VIEW`, а API требует `CATALOGS:READ`. Роли OPERATOR и VIEWER имеют `DASHBOARD:VIEW`, но не `CATALOGS:READ`.

**Варианты решения:**

| Вариант | Описание | Плюсы | Минусы |
|---------|----------|-------|--------|
| A | Добавить `CATALOGS:READ` ролям OPERATOR и VIEWER | Простая миграция | Расширяет доступ к справочникам |
| B | Создать отдельную permission `DASHBOARD:READ` для API | Гранулярный контроль | Новая миграция, дублирование |
| C | Использовать `DASHBOARD:VIEW` вместо `CATALOGS:READ` в DashboardController | Согласованность маршрут↔API | Нужно менять код контроллера |
| **D (рекомендуемый)** | Вариант C + убрать проверку `CATALOGS:READ` только из DashboardController, оставив в CatalogController | Dashboard доступен всем с `DASHBOARD:VIEW`; управление каталогами — только `CATALOGS:READ/MANAGE` | Минимальные изменения кода |

#### R-4: Добавить empty state на DashboardPage

Если тенант новый (0 устройств, 0 записей), показывать onboarding-блок:


**Условие показа:** `metrics.total_devices === 0 && metrics.tokens_total === 0`

#### R-5: Graceful degradation при 403

Если API возвращает 403 для конкретного виджета, вместо красного блока ошибки показывать серый блок «Нет доступа. Обратитесь к администратору» — без кнопки «Повторить» (retry на 403 бессмысленна).

## 9. Заключение

**Основная причина** — миграция V31 не применена на сервере. Она добавляет:
1. Permissions `CATALOGS:READ` / `CATALOGS:MANAGE` в auth-service
2. Таблицы `app_groups`, `app_group_items`, `app_aliases` в ingest-gateway

Без неё **все** эндпоинты дашборда возвращают 403 (нет permission в JWT) или 500 (таблицы не существуют).

**Дополнительная причина** — несогласованность permissions между маршрутом (`DASHBOARD:VIEW`) и API (`CATALOGS:READ`), из-за которой роли OPERATOR и VIEWER видят страницу, но не могут загрузить данные.

**Действия:**
1. **Срочно:** Применить V31 + V32 миграции на test → проверить → применить на prod
2. **Рекомендуется:** Изменить DashboardController на проверку `DASHBOARD:VIEW` вместо `CATALOGS:READ`
3. **Желательно:** Добавить onboarding empty state для новых тенантов
