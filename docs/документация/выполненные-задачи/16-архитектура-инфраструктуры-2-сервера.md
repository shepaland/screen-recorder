## Ограничения железа (актуальные при 16 Gi)

### 1. RAM (16 Gi) — ДОСТАТОЧНО для полного стека

С 16 Gi RAM ограничение по памяти **снято**:
- Полный стек (10.6 Gi) + page cache (5.4 Gi) — комфортно на одном сервере
- При split по двум серверам — есть запас для масштабирования реплик
- OpenSearch можно разместить полноценно (2-3 Gi heap) без компромиссов

### 2. HDD (200 Gi) — ОСТАЁТСЯ ГЛАВНЫМ BOTTLENECK

Расчёт объёма видеоданных:
- 1 агент: ~2-5 MB/мин (H.264 720p) = ~120-300 MB/час
- 10 000 агентов × 8 часов = **960 GB — 2.4 TB в день**
- 200 GB HDD заполнится за **2-4 часа**

**Решения (по приоритету):**
1. **Внешний S3** (Yandex Object Storage) — ~1.8 ₽/GB/мес. MinIO как прокси-кеш
2. **Retention policy:** автоудаление сегментов старше N дней (MinIO ILM)
3. **NAS / сетевое хранилище** с TB-дисками по NFS
4. **Tiered storage:** горячие сегменты (7 дней) на локальном диске, остальное во внешний S3

### 3. HDD vs SSD — производительность I/O

| Компонент | HDD impact | Влияние на 10K агентов |
|-----------|-----------|------------------------|
| PostgreSQL | Random I/O деградация при >500 concurrent queries | ⚠️ Значительное — write-heavy ingest |
| OpenSearch | Merge + flush тормозят индексацию | ⚠️ Индексация отстаёт от потока |
| Kafka | Sequential I/O — HDD допустим | ✅ Минимальное |
| MinIO | Upload latency 5-20ms вместо <1ms | ⚠️ Увеличивает время presign-upload цикла |

**Рекомендация:** SSD хотя бы для PostgreSQL data directory. Это даст 10-50× улучшение random I/O.

### 4. Сеть — пропускная способность

| Нагрузка | Трафик | 1 Gbps NIC |
|----------|--------|------------|
| 1 000 агентов | ~50 MB/s | ✅ OK |
| 3 000 агентов | ~150 MB/s | ⚠️ На пределе (125 MB/s теоретический макс) |
| 5 000 агентов | ~250 MB/s | ❌ Нужен 2.5G/10G |
| 10 000 агентов | ~500 MB/s | ❌ Нужен 10G |

### 5. CPU (8 cores) — достаточно с оговорками

| Компонент | CPU при 10K агентов | Комментарий |
|-----------|---------------------|-------------|
| ingest-gateway | ~2-3 cores | Presign + upload confirm + Kafka produce |

| OpenSearch | ~1-2 cores | Indexing + search |
| auth-service | ~0.5 core | JWT validation |
| Остальные | ~1 core | control-plane, playback, search, web |
| **Итого** | **~7-10 cores** | ⚠️ На пределе при пиках |

## Итоговая рекомендация

### Оптимальный выбор: Вариант C (Primary + Hot Standby)

**Почему:**
1. **Write/read split** — ingest на сервере 1, playback/search на сервере 2 → нет конкуренции за ресурсы
2. **OpenSearch 2-node** — поиск не теряется при падении одного сервера
3. **PostgreSQL streaming replication** — RPO ≈ 0, failover за 30-60 сек
4. **6.5 Gi резерв** на сервере 2 — достаточно для подъёма всех сервисов при failover
5. **Средняя сложность** — не требует Patroni (ручной promote достаточен), Kafka остаётся single-node

### Реалистичная оценка нагрузки

| Железо | Макс. агентов | Bottleneck |
|--------|--------------|------------|
| 2 × 8CPU/16GB/200GB HDD, 1Gbps | **~3 000** | Сеть (1 Gbps) и диск (200 Gi) |
| 2 × 8CPU/16GB/200GB SSD, 1Gbps | **~3 000** | Сеть (1 Gbps) |
| 2 × 8CPU/16GB/1TB SSD, 1Gbps | **~3 000** | Сеть (1 Gbps) |
| 2 × 8CPU/16GB/1TB SSD, 10Gbps | **~5 000** | CPU (8 cores) |
| 2 × 16CPU/32GB/1TB SSD, 10Gbps | **~10 000** | Полная конфигурация |
| + внешний S3 (Yandex OS) | **10 000+** | Без ограничений по хранению |

### План действий

1. **Сейчас:** применить Вариант C с текущими дисками, настроить PG streaming replication
2. **Краткосрочно:** подключить внешний S3 (Yandex Object Storage) для сегментов — снять ограничение 200 Gi
3. **Среднесрочно:** заменить HDD → SSD (хотя бы для PostgreSQL)
4. **При росте >3000 агентов:** апгрейд сети до 10 Gbps или балансировка ingest через CDN/edge
