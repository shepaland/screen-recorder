## Часть 1: Баг — описание проблемы

После обновления агента до v2026.3.19.1 на AIR911D (192.168.1.135) — веб-UI показывает "Онлайн" вместо "Запись идёт".

### Хронология из логов

```
22:09:35  Рестарт агента (установка новой версии)
22:09:36  State: configuring → online
22:11:38  START_RECORDING от сервера → FFmpeg started PID=27596
22:11:41  State: online → recording ✓

22:42:17  FFmpeg CRASH (attempt 1/5), session ended
22:42:17  FFmpeg restart → PID=7884
22:42:19  ❌ FFmpeg PID=7884 EXITED IMMEDIATELY
          System.InvalidOperationException: Process was not started by this object
22:42:19  State: recording → online ← ПРОБЛЕМА: агент остался "онлайн"

23:35:48  Ещё один рестарт → configuring → online
23:35:48  ❌ "AutoStart is disabled, waiting for server command"
          (сохранённый ServerConfig имеет AutoStart=false)
```

### FFmpeg stderr

```
[in#0/gdigrab] Failed to capture image (error 5)
[out#0/segment] Output file does not contain any stream
Error opening output files: Invalid argument
```

**Error 5 = ERROR_ACCESS_DENIED** — gdigrab не может захватить desktop. Причина: RDP-сессия отключена (пользователь закрыл RDP, но не вышел из системы). Desktop недоступен для захвата.

## Часть 2: Корневые причины (3 штуки)

### Причина 1: FFmpeg не может работать без активного desktop

Windows Service запускает FFmpeg через `CreateProcessAsUser()` в пользовательской сессии. Когда пользователь **отключает RDP** (но не логаутится) — desktop-сессия переходит в "disconnected" состояние, gdigrab получает error 5.

**Затронутые файлы:**
- `Capture/FfmpegProcess.cs` — `StartInUserSession()`: `CreateProcessAsUser` не проверяет доступность desktop
- `Capture/ScreenCaptureManager.cs:104-114` — `Start()`: при error ставит `_isRecording = false`, но нет специфической обработки "нет desktop"

### Причина 2: После crash retry failure — нет повторных попыток auto-start

Цепочка событий:
1. FFmpeg crash → попытка рестарта → FFmpeg "exited immediately" (нет desktop)
2. `CommandHandler.CheckAndRecoverAsync()`: `_consecutiveFailures = 1`, backoff 30s
3. `AgentService.SyncStateWithRecording()`: `!IsRecording && !HasCrashed` → `SetState(Online)`
4. Агент в состоянии `Online`, НО:
   - `HeartbeatService` видит `CurrentState == Online` и пытается auto-start
   - `AutoStartRecordingAsync()` вызывается... **но FFmpeg опять не стартует** (нет desktop)
   - **Нет backoff** для auto-start retry в HeartbeatService — каждые 30s попытка и fail
   - В логах: только одна попытка видна, потом тишина

**Затронутые файлы:**
- `Command/HeartbeatService.cs:129-146` — auto-start retry без backoff и без лимита
- `Command/CommandHandler.cs:220-257` — crash recovery: 5 попыток с exponential backoff, но привязан к `HasCrashed` (FFmpeg alive = не crash)

### Причина 3: Saved ServerConfig с AutoStart=false при рестарте

При рестарте в 23:35 — сохранённый на диск ServerConfig содержит `AutoStart=false`.

`CommandHandler.AutoStartRecordingAsync()` строка 195:
```csharp
var autoStart = serverCfg?.AutoStart ?? _config.Value.AutoStart;
```

Если `serverCfg.AutoStart == false` (не null, а именно false) — fallback на appsettings не срабатывает.

**Затронутые файлы:**
- `Command/CommandHandler.cs:195` — `??` operator не спасает от `false`
- `Auth/AuthManager.cs:321` — `merged.ConfigReceivedFromServer = existing || incoming` — сохраняет старое значение AutoStart при merge

## Часть 3: Текущая state machine агента

### Состояния (9 шт.)

```
Starting → Configuring → AwaitingUser ─┐
                       → Online ────────┤
                       → Error          │
                                        ▼
                             ┌─────── Online ◄────────────────┐
                             │          │ ▲                    │
                             │          │ │ STOP_RECORDING     │
                             │          ▼ │                    │
                             │       Recording ──► Idle        │
                             │          │      (lock/logoff)   │
                             │          ▼                      │
                             │        Error ───────────────────┘
                             │      (crash, 5 retries)
                             ▼
                      RecordingDisabled
                      (recording_enabled=false)
                             │
                             └──► Online (recording_enabled=true)

                        Any ──► Stopped (shutdown)
```

### Маппинг состояний агент → сервер → UI

| AgentState | Heartbeat status | Сервер (Device.status) | UI |
|------------|-----------------|----------------------|------|
| Starting | `starting` | starting | — |
| Configuring | `configuring` | configuring | — |
| AwaitingUser | `awaiting_user` | awaiting_user | "Ожидание" |
| **Online** | **`online`** | **online** | **"Онлайн" (жёлтый)** |
| **RecordingDisabled** | **`online`** (!) | **online** | **"Онлайн"** |
| **Recording** | **`recording`** | **recording** | **"Запись" (зелёный)** |
| Idle | `idle` | idle | "Пауза" |
| Error | `error` | error | "Ошибка" (красный) |
| Stopped | `stopped` | stopped/offline | "Офлайн" |

**Проблема:** `RecordingDisabled` отправляется как `online` — сервер не знает, что запись запрещена. UI не различает "онлайн, запись возможна" и "онлайн, запись заблокирована".

## Часть 4: Предложения по оптимизации до промышленного уровня

### 4.1 [Agent + Server] Новый heartbeat status: `recording_disabled`

**Проблема:** RecordingDisabled маппится в `online` → сервер и UI не различают.

**Решение:**
- **Агент:** `ToHeartbeatStatus()` для `RecordingDisabled` → `"recording_disabled"` (не `"online"`)
- **Сервер:** Добавить `recording_disabled` в CHECK constraint таблицы devices
- **Фронт:** DeviceStatusBadge — новый бейдж "Запись отключена" (серый)

### 4.2 [Agent] Graceful degradation при отсутствии desktop

**Проблема:** FFmpeg не может захватить экран когда RDP disconnected → crash loop.

**Решение — новое состояние `DesktopUnavailable`:**
```
Recording ──► DesktopUnavailable (gdigrab error 5)
                    │
                    │ (poll каждые 30s: desktop доступен?)
                    │
                    ▼
              Recording (desktop вернулся)
```

**Реализация:**
- `ScreenCaptureManager.Start()`: при error 5 → return специальный код `DesktopUnavailable`
- `AgentService`: новое состояние `DesktopUnavailable`, heartbeat отправляет `"desktop_unavailable"`
- **Не тратить retry на FFmpeg** — просто ждать пока desktop появится
- `SessionWatcher`: при `RemoteConnect`/`ConsoleConnect` → автоматически перезапустить запись
- **Сервер:** принимать `desktop_unavailable` как валидный статус, UI: "Нет рабочего стола" (оранжевый)

### 4.3 [Agent] Auto-start retry с exponential backoff и cooldown

**Проблема:** HeartbeatService пытается auto-start каждые 30s без backoff → спам ошибками.

**Решение:**
```csharp
// HeartbeatService
private DateTime _lastAutoStartAttempt;
private int _autoStartFailures;
private static readonly int[] BackoffSeconds = { 30, 60, 120, 300, 600 }; // max 10 min

if (shouldAutoStart)
{
    var backoff = BackoffSeconds[Math.Min(_autoStartFailures, BackoffSeconds.Length - 1)];
    if ((DateTime.UtcNow - _lastAutoStartAttempt).TotalSeconds >= backoff)
    {
        _lastAutoStartAttempt = DateTime.UtcNow;
        var started = await AutoStartRecordingAsync(...);
        if (started) _autoStartFailures = 0;
        else _autoStartFailures++;
    }
}
```

### 4.4 [Agent] Не доверять saved AutoStart при boot

**Проблема:** Сохранённый ServerConfig может содержать устаревший `AutoStart=false`.

**Решение:**
- При boot: **игнорировать saved AutoStart**, ждать первого heartbeat от сервера
- Или: при boot всегда выставлять `ConfigReceivedFromServer = false` → auto-start отложен до первого heartbeat
- HeartbeatService при первом ответе: принудительно вызвать auto-start

### 4.5 [Server] Серверный state machine для устройства

**Проблема:** Сервер — stateless зеркало агента. Если агент завис — сервер вечно показывает последний статус.

**Решение — серверные таймауты:**
```java
// DeviceService.java
@Scheduled(fixedRate = 60_000) // каждую минуту
public void checkDeviceTimeouts() {
    // Если heartbeat не приходил > 90s → status = "offline"
    deviceRepo.updateStatusForStaleDevices(
        "offline", Duration.ofSeconds(90));

    // Если heartbeat не приходил > 5min → is_active = false
    deviceRepo.deactivateStaleDevices(Duration.ofMinutes(5));
}
```

**Таймауты:**
| Пропуск heartbeat | Действие сервера |
|-------------------|-----------------|
| > 90 секунд | status → `offline` |
| > 5 минут | is_active → false |
| > 24 часа | Алерт в dashboard |

### 4.6 [Server] Heartbeat response: `desired_state`

**Проблема:** Сервер может только отправлять команды. Нет механизма "я хочу чтобы ты записывал".

**Решение — добавить `desired_state` в heartbeat response:**
```json
{
  "desired_state": "recording",  // или "online", "stopped"
  "upload_enabled": true,
  "device_settings": { ... }
}
```

Агент сравнивает `desired_state` с `CurrentState`:
- Если `desired_state = recording` и `CurrentState = online` → auto-start
- Если `desired_state = online` и `CurrentState = recording` → stop
- Это **декларативное** управление вместо **императивного** (команды)

**Преимущества:**
- Сервер не нужно знать текущее состояние агента для отправки команды
- Idempotent по определению — повторный heartbeat с тем же desired_state безопасен
- При crash/restart агент сам приведёт себя в desired_state
- Нет race conditions с командами (команда может прийти позже, чем нужно)

### 4.7 [Agent] Watchdog для FFmpeg процесса

**Проблема:** FFmpeg crash обнаруживается через polling каждые 30 секунд (maintenance loop). До 30 секунд потери записи.

**Решение — Process.Exited event:**
```csharp
_ffmpeg.Process.EnableRaisingEvents = true;
_ffmpeg.Process.Exited += (s, e) =>
{
    _logger.LogWarning("FFmpeg exited unexpectedly, PID={Pid}", _ffmpeg.Process.Id);
    // Немедленно начать recovery
    _ = Task.Run(() => OnFfmpegCrashAsync());
};
```

Это сокращает время обнаружения crash с ~30s до ~0s.

### 4.8 [Agent + Server] Health score вместо бинарного статуса

**Проблема:** Статус — грубый индикатор. "Онлайн" не говорит о проблемах.

**Решение — health score в heartbeat metrics:**
```json
{
  "health_score": 85,
  "health_issues": [
    "ffmpeg_restart_count_high",
    "segments_pending_backlog",
    "disk_space_low"
  ]
}
```

**Формула:**
```
100 (базовый)
- 20 если FFmpeg restarts > 2 за час
- 15 если segments_pending > 50
- 10 если disk_free < 1GB
- 10 если activity_pending > 1000
- 5 если CPU > 80%
```

Dashboard: цветовой индикатор рядом со статусом (зелёный/жёлтый/красный).

### 4.9 [Server] Аналитика статусов устройств

**Использовать таблицу `device_status_log`:**
- Dashboard виджет: "Время в каждом статусе за последние 24ч"
- Алерты: устройство > 1ч в Error
- Отчёт: uptime записи = time(recording) / time(online + recording)

## Часть 7: Воспроизведение бага

1. Подключиться к AIR911D по RDP
2. Убедиться запись идёт (status: recording)
3. **Отключить RDP** (закрыть окно, НЕ "Выйти")
4. Подождать ~1 мин → FFmpeg crash (gdigrab error 5)
5. Web-UI: статус "Онлайн" вместо "Запись"
