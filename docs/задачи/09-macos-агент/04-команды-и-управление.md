# Фаза 4: Команды и управление

## Цель

Реализовать обработку серверных команд, получаемых через heartbeat: START/STOP_RECORDING, UPDATE_SETTINGS, RESTART_AGENT, UPLOAD_LOGS.

## Зависимости

- Фаза 1 (heartbeat)

## Задачи

1. Реализовать `CommandHandler` — диспетчер команд из heartbeat response
2. Обработка каждого типа команд
3. ACK (подтверждение) успеха/неудачи каждой команды
4. Загрузка логов по команде UPLOAD_LOGS

## Команды (из heartbeat `pending_commands`)

Каждая команда в массиве `pending_commands` имеет формат:

```json
{
  "id": "cmd-uuid",
  "command_type": "START_RECORDING",
  "payload": {},
  "created_ts": "2026-03-17T10:30:00Z"
}
```

### Типы команд

| Команда | Payload | Действие агента |
|---------|---------|-----------------|
| `START_RECORDING` | `{}` | Создать сессию → запустить ScreenCaptureKit |
| `STOP_RECORDING` | `{}` | Остановить захват → финализировать сегмент → закрыть сессию |
| `UPDATE_SETTINGS` | `{ capture_fps, resolution, quality, segment_duration_sec, auto_start, session_max_duration_min }` | Применить новые настройки (если идёт запись — перезапустить с новыми параметрами) |
| `RESTART_AGENT` | `{}` | Graceful restart: остановить запись → exit → LaunchAgent перезапустит |
| `UPLOAD_LOGS` | `{}` | Собрать логи → POST на сервер |

## API-эндпоинты

### 4.1 Подтверждение команды (ACK)

```
PUT /api/cp/v1/devices/commands/{commandId}/ack
```

**Заголовки:**
```
Authorization: Bearer <access_token>
X-Device-ID: <device_id>
Content-Type: application/json
```

**Request (успех):**
```json
{
  "status": "acknowledged"
}
```

**Request (ошибка):**
```json
{
  "status": "failed",
  "result": {
    "message": "Screen Recording permission not granted"
  }
}
```

**Response:** 204 No Content

---

### 4.2 Загрузка логов

```
POST /api/cp/v1/devices/{deviceId}/logs
```

**Заголовки:**
```
Authorization: Bearer <access_token>
X-Device-ID: <device_id>
Content-Type: application/json
```

**Request:**
```json
[
  {
    "log_type": "kadero-agent",
    "content": "... содержимое лог-файла (tail, max 500KB) ...",
    "from_ts": "2026-03-17T10:00:00Z",
    "to_ts": "2026-03-17T10:30:00Z"
  },
  {
    "log_type": "kadero-http",
    "content": "...",
    "from_ts": "2026-03-17T10:00:00Z",
    "to_ts": "2026-03-17T10:30:00Z"
  }
]
```

**Response:** 200 OK

## Логика обработки

```
Heartbeat response → pending_commands[]
  ├── Для каждой команды:
  │   ├── Выполнить действие
  │   ├── PUT /commands/{id}/ack (status: acknowledged)
  │   └── При ошибке: PUT /commands/{id}/ack (status: failed, message)
  └── Команды обрабатываются последовательно
```

## Применение device_settings из heartbeat

Помимо команд, heartbeat возвращает `device_settings`. Агент должен:

1. Сравнить с текущими настройками
2. При изменении — обновить локальную конфигурацию
3. Если изменились параметры записи (fps, resolution, quality) и запись идёт — перезапустить запись с новыми параметрами
4. Если `recording_enabled = false` — остановить запись, статус `recording_disabled`
5. Если `auto_start = true` и запись не идёт — начать запись

## macOS-специфика

- **Логи**: `~/Library/Logs/Kadero/kadero-agent.log`, `~/Library/Logs/Kadero/kadero-http.log`
- **Restart**: `exit(0)` при наличии LaunchAgent (KeepAlive=true) — система перезапустит
- **Размер логов**: tail последних 500KB каждого файла

## Критерии приёмки

1. START_RECORDING → запись стартует, ACK отправляется
2. STOP_RECORDING → запись останавливается корректно, ACK отправляется
3. UPDATE_SETTINGS → настройки применяются (fps, resolution обновляются)
4. RESTART_AGENT → процесс перезапускается gracefully
5. UPLOAD_LOGS → логи загружаются на сервер, ACK отправляется
6. При ошибке выполнения — ACK с status=failed и описанием ошибки
7. device_settings из heartbeat корректно применяются
