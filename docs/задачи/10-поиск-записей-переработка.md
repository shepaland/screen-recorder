# 10. Переработка страницы "Поиск записей"

## Текущее состояние

Страница `/search` (SearchPage.tsx) — минимальная реализация:
- Полнотекстовый поиск через OpenSearch (search-service)
- Результаты — плоский список **сегментов** (не сессий)
- Клик по сегменту — навигация на отдельную страницу `/recordings/{sessionId}` (SessionDetailPage)
- Фильтры: только текстовый запрос
- Нет группировки по сессиям, нет inline-плеера

### Проблемы текущей версии

1. **Сегменты вместо сессий** — пользователь видит десятки строк одной сессии вместо одной
2. **Нет фильтрации** по сотруднику, устройству, дате, размеру
3. **Переход на отдельную страницу** — нет модального плеера, теряется контекст поиска
4. **Нет информации о сотруднике** — ни в сегментах OpenSearch, ни в UI
5. **Нет раскрытия сегментов** — нельзя увидеть состав сессии без перехода

## Требования

### Основной интерфейс — список сессий

Таблица с колонками:

| Колонка | Источник | Примечание |
|---------|----------|------------|
| Сотрудник | `recording_sessions.user_id` → `users.username` (или `first_name + last_name`) | Требует JOIN или enrichment |
| Устройство | `recording_sessions.device_id` → `devices.hostname` | Уже есть в `RecordingListItemResponse.deviceHostname` |
| Сессия | `recording_sessions.id` | UUID, показывать короткий формат |
| Дата/время начала | `recording_sessions.started_ts` | Формат: `dd.MM.yyyy HH:mm:ss` |
| Дата/время окончания | `recording_sessions.ended_ts` | `null` → "В процессе" |
| Кол-во сегментов | `recording_sessions.segment_count` | Доступных к просмотру (status=confirmed) |
| Размер (МБ) | `recording_sessions.total_bytes` | Конвертация в МБ на фронте |

### Гармоника (раскрытие сессии)

При клике на строку сессии — раскрывается аккордеон вниз с таблицей сегментов:

| Колонка сегмента | Источник |
|------------------|----------|
| # (порядковый) | `segment.sequence_num` |
| Время записи | `segment.recorded_at` или `segment.created_ts` |
| Длительность | `segment.duration_ms` → формат `mm:ss` |
| Размер | `segment.size_bytes` → МБ |
| Статус | `segment.status` |
| Действие | Кнопки: "Воспроизвести" + "Скачать" (MP4) |

### Модальное окно с плеером

**Два режима открытия:**

1. **Воспроизведение всей сессии** — кнопка "Смотреть" на строке сессии:
   - Модальное окно с HLS-плеером
   - Загрузка M3U8 плейлиста всей сессии: `/api/playback/v1/playback/sessions/{sessionId}/playlist.m3u8`
   - Под видео — полоса сегментов с возможностью переключения
   - Клик по сегменту → перемотка на соответствующий offset в потоке

2. **Воспроизведение с конкретного сегмента** — кнопка "Воспроизвести" в строке сегмента:
   - Открывается то же модальное окно
   - Воспроизведение начинается с выбранного сегмента
   - Доступно переключение на любой другой сегмент сессии

### Выгрузка (скачивание)

Выгрузка доступна в **двух местах**: в таблице сессий/сегментов и в модальном окне плеера.

#### A. Выгрузка целой сессии (ZIP-архив)

**Точки вызова:**
1. Кнопка "Скачать" (иконка архива) на строке сессии в таблице
2. Кнопка "Скачать сессию" в модальном окне плеера (toolbar рядом с "Закрыть")

**Механика:**
- Используется существующий эндпоинт: `GET /api/v1/ingest/recordings/{id}/download?format=zip`
- Формат ответа: `application/zip` с `Content-Disposition: attachment`
- Структура архива:
  ```
  session_{short_id}/
  ├── recording.json          # метаданные: session_id, device, timestamps, segments[]
  ├── 00001.mp4               # сегмент 1
  ├── 00002.mp4               # сегмент 2
  └── ...
  ```
- Имя файла: `{hostname}_{date}_{short_session_id}.zip`
- Бэкенд уже реализован: `RecordingController.downloadRecording()` — стримит ZIP с сегментами из S3
- Требуется пермишен `RECORDINGS:EXPORT`
- Active сессии (status=active) не могут быть скачаны — показывать tooltip "Запись в процессе"

#### B. Выгрузка отдельного сегмента (MP4)

**Точки вызова:**
1. Кнопка "Скачать" (иконка скачивания) на строке сегмента в раскрытой гармошке
2. Кнопка "Скачать сегмент" в модальном окне плеера (рядом с текущим активным сегментом в SegmentTimeline)

**Механика:**
- Используется существующий эндпоинт: `GET /api/v1/ingest/recordings/{sessionId}/download?format=mp4`
  - Текущая реализация поддерживает только single-segment записи для MP4
  - **Нужна доработка**: новый эндпоинт для скачивания конкретного сегмента

**Новый эндпоинт (бэкенд):**
```
GET /api/v1/ingest/recordings/{sessionId}/segments/{segmentId}/download
```
- Возвращает: `Content-Type: video/mp4`, `Content-Disposition: attachment; filename="{hostname}_{date}_seg{seq_num}.mp4"`
- Стримит содержимое сегмента напрямую из S3 (`s3Service.getObject(segment.getS3Key())`)
- Валидация: сегмент принадлежит сессии, сессия принадлежит тенанту
- Требуется пермишен `RECORDINGS:EXPORT`

#### C. Состояния UI для кнопок выгрузки

| Состояние сессии | Скачать сессию (ZIP) | Скачать сегмент (MP4) |
|------------------|---------------------|-----------------------|
| `active` | Заблокировано, tooltip "Запись в процессе" | Доступно (для confirmed сегментов) |
| `completed` | Доступно | Доступно |
| `interrupted` | Доступно | Доступно |
| `failed` | Доступно (что записано) | Доступно (для confirmed сегментов) |

| Статус сегмента | Скачать | Воспроизвести |
|-----------------|---------|---------------|
| `confirmed` | Доступно | Доступно |
| `uploaded` | Заблокировано | Заблокировано |
| `failed` | Заблокировано | Заблокировано |

Фильтры над таблицей:

| Фильтр | Тип | API-параметр |
|--------|-----|-------------|
| Сотрудник | Текстовый поиск / dropdown | `employee_name` (новый) |

## Архитектура решения

### Источник данных — ingest-gateway (НЕ search-service)

Текущий поиск идёт через OpenSearch (search-service), но для нового интерфейса правильнее использовать **ingest-gateway** `GET /api/v1/ingest/recordings`:

- Уже возвращает `RecordingListItemResponse` с `deviceHostname`, `segmentCount`, `totalBytes`
- Поддерживает фильтры: `status`, `device_id`, `from`, `to`, пагинация
- Данные из PostgreSQL — consistent, не зависит от задержки индексации OpenSearch
- OpenSearch поиск оставить как **дополнительную** функцию полнотекстового поиска (по метаданным, OCR и т.п.)

### Необходимые доработки бэкенда (ingest-gateway)

#### 1. Enrichment: имя сотрудника

`RecordingListItemResponse` сейчас не содержит имя пользователя. Нужно:

- В `RecordingSession` уже есть `user_id`
- Добавить JOIN с `users` (через `user_id`) для получения `username` / `first_name` / `last_name`
- Добавить поле `employeeName` (или `username`) в `RecordingListItemResponse`

**Вариант A** — JOIN в SQL запросе `findByTenantIdWithFilters`:
```sql
SELECT rs.*, d.hostname as device_hostname, u.username as employee_name
FROM recording_sessions rs
LEFT JOIN devices d ON d.id = rs.device_id
LEFT JOIN users u ON u.id = rs.user_id
WHERE rs.tenant_id = :tenantId
  AND (:status IS NULL OR rs.status = :status)
  ...
```

**Вариант B** — Enrichment в сервисном слое (batch-запрос user_ids).

Рекомендация: **Вариант A** — проще, один запрос.

#### 2. Дополнительные фильтры

Расширить `RecordingService.listRecordings()`:
- `employee_name` — `ILIKE '%name%'` по `u.username` или `u.first_name || ' ' || u.last_name`
- `min_segments` / `max_segments` — `rs.segment_count >= :min AND rs.segment_count <= :max`
- `min_bytes` / `max_bytes` — `rs.total_bytes >= :min AND rs.total_bytes <= :max`
- Текстовый `search` — `d.hostname ILIKE '%q%' OR u.username ILIKE '%q%' OR CAST(rs.id AS TEXT) LIKE '%q%'`

#### 3. Подсчёт confirmed-сегментов

`recording_sessions.segment_count` считает все сегменты. Для "доступных к просмотру" нужно:
- Либо отдельное поле `confirmed_segment_count`
- Либо при запросе сегментов считать `WHERE status = 'confirmed'`

Рекомендация: использовать `segment_count` как есть (все сегменты ≈ confirmed для завершённых сессий). Для точности — добавить `confirmed_segment_count` в DTO, вычислять через subquery.

#### 1. Новая страница SearchPage (переписать)

```
    ├── VideoPlayer (HLS.js)

```

#### 2. Новый компонент PlayerModal

- Модальное окно (overlay + centered container)
- Принимает: `sessionId`, `segments[]`, `initialSegmentIndex` (опционально)
- Содержит: `VideoPlayer` + полоса сегментов
- Переключение сегмента: пересоздание HLS-плеера с новым offset или навигация по playlist

**Механика переключения сегментов в HLS:**

HLS.js загружает весь M3U8 playlist. Каждый `#EXTINF` — это сегмент. Чтобы начать с конкретного сегмента:
- Вариант A: `hls.startLoad(startPosition)` — установить начальную позицию в секундах (сумма длительностей предыдущих сегментов)
- Вариант B: `video.currentTime = offset` после `MANIFEST_PARSED`

Рекомендация: **Вариант B** — проще и надёжнее.

#### 3. API-клиент

Новая функция для загрузки списка сессий:
```typescript

```

Существующая функция `getRecordingSegments(id)` — уже есть, используется для гармоники.

## Риски

1. **Имя сотрудника**: `user_id` в `recording_sessions` может быть `null` (устройство без привязки к пользователю). Нужен fallback — показывать "—" или hostname.
2. **HLS seek**: переход к конкретному сегменту через `currentTime` может быть неточным при разной длительности сегментов. Нужно суммировать duration предыдущих.
3. **Производительность**: JOIN с `users` на больших объёмах. Смягчение — пагинация (max 20-50 сессий на страницу).
4. **Выгрузка больших сессий**: ZIP-архив формируется на лету (стримится). Для сессий с 100+ сегментами (~6 ГБ+) может быть долгий download. Смягчение — прогресс-бар на фронте невозможен (стриминг), но можно показать размер сессии перед скачиванием + подтверждение для сессий > 1 ГБ.
5. **Права доступа**: выгрузка требует `RECORDINGS:EXPORT`, воспроизведение — `RECORDINGS:READ`. Кнопки скачивания должны скрываться при отсутствии пермишена.

## Тест-кейсы

### TC-S01: Отображение списка сессий

| Поле | Значение |
|------|----------|
| Предусловия | Пользователь авторизован как `maksim` (OWNER), тенант "Тест 1". Есть минимум 2 завершённые сессии с сегментами. |
| Шаги | 1. Перейти на страницу "Поиск записей" (/search) |
| Ожидаемый результат | Таблица сессий отображается. Колонки: Сотрудник, Устройство, Сессия (короткий UUID), Начало, Конец, Сегменты (число), Размер (МБ). Данные соответствуют реальным сессиям. |

### TC-S02: Раскрытие гармошки сегментов

| Поле | Значение |
|------|----------|
| Предусловия | Страница "Поиск записей" открыта, есть сессия с >= 3 сегментами. |
| Шаги | 1. Кликнуть по строке сессии<br>2. Проверить раскрытую таблицу сегментов<br>3. Кликнуть по строке сессии повторно |
| Ожидаемый результат | 1. Под строкой сессии раскрывается таблица сегментов (аккордеон вниз)<br>2. Сегменты отображаются с колонками: #, Время записи, Длительность (mm:ss), Размер (МБ), Статус, кнопки ▶ и ⬇<br>3. Гармошка сворачивается |

### TC-S03: Воспроизведение целой сессии

| Поле | Значение |
|------|----------|
| Предусловия | Страница "Поиск записей", есть завершённая сессия с >= 2 confirmed-сегментами. |
| Шаги | 1. Нажать кнопку "Воспроизвести" (▶) на строке сессии<br>2. Дождаться загрузки плеера<br>3. Проверить полосу сегментов |
| Ожидаемый результат | 1. Открывается модальное окно с видеоплеером<br>2. HLS-видео начинает воспроизведение с первого сегмента<br>3. Под видео — полоса сегментов, первый сегмент выделен |

### TC-S04: Воспроизведение с конкретного сегмента

| Поле | Значение |
|------|----------|
| Предусловия | Гармошка сессии раскрыта, видны сегменты. |
| Шаги | 1. Нажать кнопку "Воспроизвести" (▶) на 3-м сегменте<br>2. Дождаться загрузки плеера<br>3. Проверить позицию воспроизведения |
| Ожидаемый результат | 1. Открывается модальное окно плеера<br>2. Воспроизведение начинается с 3-го сегмента (не с начала)<br>3. В полосе сегментов 3-й сегмент выделен |

### TC-S05: Переключение сегмента в плеере

| Поле | Значение |
|------|----------|
| Предусловия | Модальное окно плеера открыто, сессия с >= 3 сегментами. |
| Шаги | 1. Кликнуть по 5-му сегменту в полосе сегментов<br>2. Дождаться переключения |
| Ожидаемый результат | Воспроизведение перемотано на начало 5-го сегмента. 5-й сегмент выделен в полосе. |

### TC-S06: Скачивание сессии (ZIP) из таблицы

| Поле | Значение |
|------|----------|
| Предусловия | Пользователь имеет пермишен `RECORDINGS:EXPORT`. Есть завершённая сессия. |
| Шаги | 1. Нажать кнопку "Скачать" (⬇) на строке сессии<br>2. Дождаться скачивания |
| Ожидаемый результат | Браузер скачивает ZIP-файл. Имя файла: `{hostname}_{date}_{session_id}.zip`. Архив содержит `recording.json` + файлы `00001.mp4`, `00002.mp4`, ... |

### TC-S07: Скачивание сегмента (MP4) из гармошки

| Поле | Значение |
|------|----------|
| Предусловия | Гармошка раскрыта, есть confirmed-сегмент. Пермишен `RECORDINGS:EXPORT`. |
| Шаги | 1. Нажать кнопку "Скачать" (⬇) на строке конкретного сегмента<br>2. Дождаться скачивания |
| Ожидаемый результат | Браузер скачивает MP4-файл. Имя файла: `{hostname}_{date}_seg{seq_num}.mp4`. Видео воспроизводится локально. |

### TC-S08: Скачивание сессии (ZIP) из плеера

| Поле | Значение |
|------|----------|
| Предусловия | Модальное окно плеера открыто. Пермишен `RECORDINGS:EXPORT`. |
| Шаги | 1. Нажать кнопку "Скачать сессию (ZIP)" в модальном окне |
| Ожидаемый результат | Браузер начинает скачивание ZIP-архива всей сессии. |

### TC-S09: Скачивание сегмента (MP4) из плеера

| Поле | Значение |
|------|----------|

| Ожидаемый результат | Браузер скачивает MP4-файл текущего (3-го) сегмента. |
