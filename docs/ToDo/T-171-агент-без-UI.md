# T-171: Агент: полная работоспособность без запуска UI (TrayApp)

**Тип:** Task
**Приоритет:** High
**Статус:** In Progress
**Дата анализа:** 2026-03-19

---

## Описание

Windows-агент Кадеро должен быть полностью работоспособен даже если пользователь не включил запуск UI (иконки в трее нет). Сейчас без TrayApp не собираются focus intervals и input events.

### Что работает без TrayApp

| Компонент | Работает? | Примечание |
|-----------|-----------|-----------|
| Запись видео (ScreenCaptureManager) | Да | Независим от UI |
| Heartbeat (HeartbeatService) | Да | Независим от UI |
| Загрузка сегментов (SegmentUploader) | Да | Независим от UI |
| Команды от сервера (CommandHandler) | Да | Независим от UI |
| Sync данных (DataSyncService) | Да | Независим от UI |
| **Focus Intervals** | **Некорректно** | ActiveWindowTracker в Session 0 — `GetForegroundWindow()` возвращает 0. Нет geometry, нет browser domain через UIAutomation |

| UI статус (иконка в трее) | Нет | Не критично для функциональности |

## Текущий flow установки (setup.iss)

```
[Run] последовательность:
1. icacls              → права доступа на директории
2. schtasks /delete    → удалить legacy задачу
3. reg.exe             → re-enable StartupApproved (HKCU)
4. --register          → регистрация + AutoStartHelper.EnableAutoStart()
                         → записывает HKCU\Run\KaderoAgent = "--tray"
5. sc.exe create       → создание Windows Service
6. sc.exe failure      → авто-рестарт при падении
7. sc.exe description  → описание
8. sc.exe start        → ЗАПУСК службы (запись начинается!)
────── сервис работает, запись идёт ──────
9. [postinstall]       → ☐ "Запустить Кадеро в области уведомлений"
                         → KaderoAgent.exe --tray
                         → Flags: postinstall nowait runasoriginaluser
```

### Проблема: разрыв аналитики

**Сценарий:** пользователь снимает галочку на шаге 9.

| Момент | Сервис | Focus/Input | Разрыв |
|--------|--------|-------------|--------|
| После установки | Работает, пишет видео | **НЕТ** — `--tray` не запущен | **ДА** |
| До выхода из системы | Работает | **НЕТ** | **ДА** |
| Следующий логин | Работает | HKCU\Run → `--tray` → **ДА** | Нет |

**Разрыв может длиться часы или дни** — от момента установки до следующего логина пользователя. Всё это время видео пишется, но без focus intervals и input events.

### Корневая проблема: UI и аналитика связаны

Сейчас `--tray` — это **единый пакет**: UI (иконка, меню, статус-окно) + аналитика (TrayWindowTracker, InputTracker). Галочка "Запустить в трее" управляет обоими. Пользователь думает что отключает иконку, а на деле отключает сбор аналитики.

---

## Корневая причина (Session 0)

Windows Service работает в **Session 0** — изолированная сессия для служб без desktop:

| API | Session 0 | Интерактивная сессия |
|-----|-----------|---------------------|
| `GetForegroundWindow()` | Возвращает 0 или неверный hwnd | Корректно |
| `SetWindowsHookEx(WH_MOUSE_LL)` | Не работает | Корректно |
| `SetWindowsHookEx(WH_KEYBOARD_LL)` | Не работает | Корректно |
| `GetWindowText()` | Может работать cross-session | Корректно |
| `GetLastInputInfo()` | Для Session 0, не пользователя | Корректно |
| `MonitorFromWindow()` / `GetWindowRect()` | Некорректно | Корректно |

## Проблема 2: Запись не стартует по умолчанию после установки

### Текущая цепочка AutoStart

Запись стартует **только если все 3 gate-а пройдены** (`AgentService.ExecuteAsync`, строки 111-148):

```
Gate 1: RecordingEnabled == true   ← из токена (ServerConfig.RecordingEnabled)
Gate 2: ConfigReceivedFromServer == true  ← подтверждение от сервера
Gate 3: AutoStart == true          ← ПРОБЛЕМА: по умолчанию FALSE
```

### Откуда приходит AutoStart

```
Уровень 1: auth-service application.yml (строка 86)
  prg.device.default-auto-start: ${DEVICE_AUTO_START:false}    ← DEFAULT = FALSE

Уровень 2: DeviceAuthService.java (строки 341-342)
  Boolean autoStart = deviceSettings.containsKey("auto_start")
      ? toBoolean(deviceSettings.get("auto_start"))
      : defaultAutoStart;                                       ← FALSE если нет в settings

Уровень 3: DeviceLoginResponse.ServerConfig.autoStart
  → передаётся в ответе device-login → сохраняется в credentials

Уровень 4: Agent CommandHandler.AutoStartRecordingAsync (строка 195)
  var autoStart = serverCfg?.AutoStart ?? _config.Value.AutoStart;
  AgentConfig.AutoStart = false (appsettings.json строка 18)    ← FALLBACK ТОЖЕ FALSE
```

### Что происходит после установки

```
Шаг 1: --register → AuthManager.RegisterAsync() → POST /api/v1/auth/device-login
        Ответ: ServerConfig { AutoStart: false, RecordingEnabled: true, ConfigReceivedFromServer: true }
        Сохранено в credentials на диск

Шаг 2: sc.exe start → Service запускается
        AgentService.ExecuteAsync() → AuthManager.InitializeAsync() → загрузка credentials

Шаг 3: Проверка gate-ов:
        RecordingEnabled = true       ✓
        ConfigReceivedFromServer = true  ✓
        AutoStart = false             ✗ ← БЛОКИРУЕТ ЗАПИСЬ!

Шаг 4: Запись НЕ стартует. Состояние: Online (не Recording)

Шаг 5: Heartbeat (через 30с) → получает device_settings → auto_start: false
        HeartbeatService авто-старт проверка → AutoStart = false → НЕ стартует

Шаг 6: Запись стартует ТОЛЬКО при:
        - Явной команде START_RECORDING от сервера (через heartbeat pending_commands)
        - Или ручном изменении device.settings.auto_start = true в БД
```

**Итог:** после установки запись НЕ стартует, хотя токен разрешает (`recording_enabled=true`). Нужна либо явная команда от сервера, либо ручное изменение настроек.

### Все точки автостарта записи (по коду)

В агенте **4 точки**, где запись может стартовать автоматически:

#### Точка 1: Запуск сервиса (AgentService.ExecuteAsync, строки 111-148)

```
Сервис стартует → auth → проверка gate-ов → AutoStartRecordingAsync()

Gate-ы:
  1. _currentState != AwaitingUser         (сессия активна, юзер залогинен)
  2. serverCfg.RecordingEnabled == true     (токен разрешает)
  3. serverCfg.ConfigReceivedFromServer     (конфиг подтверждён сервером)
  4. [внутри AutoStartRecordingAsync]:
     serverCfg.AutoStart == true            (автостарт включен) ← БЛОКИРУЕТ
```

**Когда срабатывает:** загрузка ОС → Windows Service стартует → auth → если юзер уже залогинен.
**Проблема:** `AutoStart=false` по умолчанию.

#### Точка 2: Heartbeat (HeartbeatService, строки 130-146)

```
Heartbeat каждые 30с → ApplyDeviceSettings() → проверка:

  1. _agentService.CurrentState == Online   (юзер залогинен, не Recording)
  2. !_captureManager.IsRecording           (ещё не пишем)
  3. _sessionManager.CurrentSessionId == null (нет активной сессии)
  4. serverCfg.ConfigReceivedFromServer     (конфиг подтверждён)
  5. serverCfg.AutoStart == true            ← БЛОКИРУЕТ
  6. serverCfg.RecordingEnabled == true     (токен разрешает)
```

**Когда срабатывает:** первый heartbeat после старта (если Точка 1 не сработала), или heartbeat с обновлённым `auto_start`.
**Проблема:** `AutoStart=false` по умолчанию.

#### Точка 3: Логин пользователя (SessionWatcher.OnSessionSwitch → SessionLogon, строки 147-216)

```
SessionLogon event → ждёт auth (до 60с) → проверка:

  1. _authManager.IsAuthenticated           (auth прошёл)
  2. serverCfg.ConfigReceivedFromServer     (конфиг подтверждён)
  3. → CommandHandler.HandleSessionLogonAsync() (строки 115-143):
     a. Сбросить _isPausedByLock
     b. Очистить stale FFmpeg state
     c. Задержка 3с (стабилизация десктопа)
     d. Проверка RecordingEnabled           (токен разрешает)
     e. → AutoStartRecordingAsync():
        - serverCfg.AutoStart == true       ← БЛОКИРУЕТ
        - RecordingEnabled == true
```

**Когда срабатывает:** пользователь логинится в Windows (после логаута, Fast User Switching, RDP).
**Проблема:** `AutoStart=false` по умолчанию. Задержка 3с перед стартом записи — ОК (ждёт DWM/explorer).

#### Точка 4: Разблокировка экрана (SessionWatcher.OnSessionSwitch → SessionUnlock, строки 118-145)

```
SessionUnlock event → CommandHandler.ResumeRecordingAsync() (строки 77-108):

  1. _isPausedByLock == true               (была пауза)
  2. RecordingEnabled == true              (токен разрешает)
  3. serverCfg.AutoStart == true           ← БЛОКИРУЕТ
  4. serverCfg.ConfigReceivedFromServer    (конфиг подтверждён)
  5. → StartRecording()
```

**Когда срабатывает:** пользователь разблокировал экран после Win+L.
**Проблема:** `AutoStart=false` по умолчанию. Также: fallback в SessionWatcher строка 136 `serverCfg?.AutoStart ?? false` — жёстко `false`, не использует AgentConfig.

#### Дополнительно: ConsoleConnect / RemoteConnect (строки 218-253)

```
ConsoleConnect/RemoteConnect → ResumeRecordingAsync()
```

Те же gate-ы что и Unlock. Срабатывает при переключении RDP-сессии или возврате к консоли.

### Проблема: AutoStart=false блокирует ВСЕ 4 точки

Без изменения дефолтов, запись **никогда не стартует автоматически** — ни при установке, ни при логине, ни при разблокировке. Единственный путь — явная команда `START_RECORDING` от сервера через heartbeat.

### Дополнительная проблема: SessionWatcher hardcoded fallback

В `SessionWatcher.cs` строка 136:
```csharp
var autoStart = serverCfg?.AutoStart ?? false;  // ← hardcoded false, не AgentConfig!
```

Даже если `AgentConfig.AutoStart = true`, SessionWatcher использует `false` как fallback. Это влияет на определение состояния (Recording vs Online) после unlock, но не на саму запись — запись зависит от `ResumeRecordingAsync()` → `CommandHandler`, который использует правильный fallback `_config.Value.AutoStart`.

Тем не менее, строка 136 должна быть исправлена для консистентности.

### Решение: AutoStart = true по умолчанию

**Принцип:** если токен разрешает запись (`recording_enabled=true`), запись должна стартовать сразу. Админ может отключить для конкретного устройства через `auto_start: false` в device.settings.

#### Изменения (4 точки):

**1. Сервер: auth-service application.yml (строка 86)**
```yaml
# Было:
default-auto-start: ${DEVICE_AUTO_START:false}
# Стало:
default-auto-start: ${DEVICE_AUTO_START:true}
```

**2. Агент: appsettings.json (строка 18)**
```json
// Было:
"AutoStart": false,
// Стало:
"AutoStart": true,
```

**3. Агент: AgentConfig.cs (строка 14)**
```csharp
// Было:
public bool AutoStart { get; set; } = false;   // Default: no auto-start without server config
// Стало:
public bool AutoStart { get; set; } = true;    // Default: auto-start when recording_enabled=true
```

**4. Агент: SessionWatcher.cs (строка 136) — fix hardcoded fallback**
```csharp
// Было:
var autoStart = serverCfg?.AutoStart ?? false;
// Стало:
var autoStart = serverCfg?.AutoStart ?? true;
```

Без этого фикса: после unlock при `ServerConfig.AutoStart == null` состояние будет Online вместо Recording, хотя запись фактически идёт (CommandHandler использует правильный fallback).

#### Результат после изменений

```
Шаг 1: --register → ServerConfig { AutoStart: true, RecordingEnabled: true, ConfigReceivedFromServer: true }
Шаг 2: Service запускается → AgentService.ExecuteAsync()
Шаг 3: Gate-ы:
        RecordingEnabled = true          ✓
        ConfigReceivedFromServer = true  ✓
        AutoStart = true                 ✓  ← ТЕПЕРЬ ПРОХОДИТ
Шаг 4: AutoStartRecordingAsync() → StartRecording() → FFmpeg запускается
Шаг 5: Состояние: Recording. Запись идёт сразу после установки!
```

#### Матрица поведения (с учётом всех gate-ов)

| recording_enabled (токен) | auto_start (server/default) | Результат |
|---|---|---|
| `true` | `true` (default) | **Запись стартует автоматически** |
| `true` | `false` (device.settings) | Запись не стартует, ждёт команду START_RECORDING |
| `false` | `true` | Запись заблокирована. Состояние: RecordingDisabled |
| `false` | `false` | Запись заблокирована. Состояние: RecordingDisabled |

#### Приоритет настроек (от высшего к низшему)

```
1. device.settings JSONB (per-device, в БД)     ← админ может перезаписать
2. ServerConfig из device-login/heartbeat        ← серверный дефолт
3. DEVICE_AUTO_START env var на сервере          ← деплой-дефолт
4. AgentConfig.AutoStart (appsettings.json)      ← локальный fallback
```

#### Обратная совместимость

- Существующие устройства с `device.settings.auto_start = false` → не затронуты
- Существующие устройства БЕЗ `auto_start` в settings → при следующем heartbeat получат `true` (новый дефолт) → запись стартует
- Если нужно НЕ стартовать на конкретном устройстве → установить `auto_start: false` в device.settings через API

#### Деплой на test-стейджинг

Для test: добавить env var `DEVICE_AUTO_START=true` в deployment auth-service.
Для prod: то же, но после подтверждения.

Без env var: дефолт в application.yml теперь `true`, так что работает из коробки.

## Решение: Вариант B (headless) — разделение UI и аналитики

### Ключевой принцип

**Два независимых процесса с разными задачами:**

| | `--headless` (аналитика) | `--tray` (UI) |
|---|---|---|
| **Назначение** | Сбор focus intervals + input events | Иконка в трее, статус, меню |
| **Запуск** | ОБЯЗАТЕЛЬНЫЙ, автоматический | ОПЦИОНАЛЬНЫЙ, по желанию |
| **Контроль** | Инсталлятор/сервис, пользователь НЕ может отключить | Пользователь решает |
| **Видимость** | Невидимый, нет иконки | Иконка в трее |
| **Содержит** | TrayWindowTracker, InputTracker, PipeClient, status poll | NotifyIcon, StatusWindow, меню |

### Целевая архитектура

```
┌────────────────────────────────────────────────────────────────┐
│  Windows Service (Session 0)   --service                        │
│  PipeServer, ScreenCapture, DataSync, Upload, Commands          │
└─────────────────────────▲──────────────────────────────────────┘
                          │ Named Pipe "KaderoAgent"
              ┌───────────┴───────────┐
              │                       │
┌─────────────┴───────────┐  ┌────────┴──────────────────────────┐
│  --headless              │  │  --tray (ОПЦИОНАЛЬНО)             │
│  ОБЯЗАТЕЛЬНЫЙ            │  │  Запускается из Start Menu        │
│  HKCU\Run (постоянный)  │  │                                   │
│  Mutex: Local\Kadero-    │  │  НЕТ аналитики (только UI)       │
│         AgentCollector   │  │  Mutex: Local\KaderoAgentTray     │
│                          │  │                                   │
│  TrayWindowTracker       │  │  NotifyIcon + ContextMenu         │
│  InputTracker            │  │  StatusWindow + AboutDialog       │
│  PipeClient (status)     │  │  PipeClient (status poll 3s)     │
│  Status poll 3s          │  │  Reconnect, Restart Service       │
│  SegmentStartTs binding  │  │                                   │
│                          │  │  НЕТ: TrayWindowTracker           │
│  Message pump (STA)      │  │  НЕТ: InputTracker                │
│  Невидимый               │  │                                   │
└──────────────────────────┘  └───────────────────────────────────┘
```

**`--headless` и `--tray` работают ПАРАЛЛЕЛЬНО** — разные mutex, разные задачи, не конфликтуют.

### Flow установки (новый)

```

1. icacls              → права доступа
2. schtasks /delete    → удалить legacy задачу
3. reg.exe             → re-enable StartupApproved
4. --register          → регистрация + AutoStartHelper.EnableAutoStart()
                         → HKCU\Run\KaderoAgent = "--headless"  (ИЗМЕНЕНО!)
5. sc.exe create       → создание Windows Service
6. sc.exe start        → ЗАПУСК службы
7. KaderoAgent.exe     → --headless (ОБЯЗАТЕЛЬНЫЙ, не postinstall)
   --headless            Flags: nowait runasoriginaluser
```

## План реализации

### 1. Новый класс `HeadlessApplication` (Tray/HeadlessApplication.cs)

Лёгкий `ApplicationContext` — аналитика из TrayApplication, без UI:

```csharp
public class HeadlessApplication : ApplicationContext
{
    private readonly PipeClient _pipeClient;
    private readonly TrayWindowTracker _windowTracker;
    private readonly InputTracker _inputTracker;
    private System.Threading.Timer? _statusTimer;
    private int _polling; // 0=idle, 1=busy
    private string _lastConnectionStatus = "";

    public HeadlessApplication()
    {
        _pipeClient = new PipeClient();
        _windowTracker = new TrayWindowTracker();
        _inputTracker = new InputTracker();

        // Status polling для получения SegmentStartTs (как TrayApplication строка 120)
        _statusTimer = new System.Threading.Timer(_ => PollStatus(), null, 0, 3000);

        _inputTracker.Start();
        _windowTracker.SetInputTracker(_inputTracker);
        _windowTracker.Start();

        // Graceful shutdown при logoff/shutdown
        SystemEvents.SessionEnding += (_, _) => Shutdown();
    }

    private void PollStatus() { /* аналог TrayApplication.PollStatusOnThreadPool без UI */ }
    private void Shutdown() { /* dispose tracker, input, pipe → Application.Exit() */ }
}
```

**Что берём из TrayApplication (строки 131-174):**
- `PollStatusOnThreadPool()` — подключение к pipe, `GetStatusAsync()`, `Interlocked` guard
- `_inputTracker.UpdateSegmentContext(status.SegmentStartTs)` (строка 155) — **КРИТИЧНО** для привязки к видео
- Таймауты: 5s first connect, 1s reconnect (строка 142)

**Что НЕ берём:**
- `NotifyIcon`, `_invokeControl`, `ContextMenuStrip`, `StatusWindow`, `AboutDialog`
- `MarshalUpdateTrayIcon()`, `UpdateTrayIcon()` — нет UI для обновления
- Обработчики меню: `OnTrayDoubleClick`, `OnReconnect`, `OnRestartService`, `OnAbout`, `OnExit`

**Message pump:**
- `Application.Run(new HeadlessApplication())` — создаёт message loop без видимого окна
- Необходим для `WH_MOUSE_LL` / `WH_KEYBOARD_LL` hooks в InputTracker
- `ApplicationContext` без формы — невидим для пользователя

## Критические детали реализации

### Почему два mutex, а не один

| Подход | --headless | --tray | Проблема |
|--------|-----------|--------|----------|
| **Один mutex** (старая аналитика) | Mutex `Local\KaderoAgentTray` | Тот же | Если headless занял mutex → tray не может стартовать. Нужен handover protocol (сигнал остановки + ожидание). Сложно |
| **Два mutex** (новая аналитика) | `Local\KaderoAgentCollector` | `Local\KaderoAgentTray` | Работают **параллельно**, не мешают друг другу. Просто |

Два mutex — проще, надёжнее, нет race conditions при handover.

### Status polling в headless (получение SegmentStartTs)

TrayApplication делает poll каждые 3с (`PollStatusOnThreadPool`, строки 131-174):
- `PipeClient.GetStatusAsync()` → `AgentStatus.SegmentStartTs`
- `_inputTracker.UpdateSegmentContext(status.SegmentStartTs)` (строка 155)

Без этого `SegmentOffsetMs` будет null → input events не привязаны к видео.

**HeadlessApplication ОБЯЗАН** реализовать тот же status polling, но без UI-обновлений.

### Message pump для hooks

`WH_MOUSE_LL` и `WH_KEYBOARD_LL` требуют message pump в потоке установки hook.
- `Application.Run(applicationContext)` без формы — создаёт message loop
- `InputTracker.Start()` вызывает `SetWindowsHookEx()` — должно быть на STA thread с message pump
- `ApplicationContext` обеспечивает `Application.Run()` без видимого окна

### TrayWindowTracker использует свой PipeClient

`TrayWindowTracker` создаёт **собственный** `PipeClient` (отдельный от status-polling PipeClient). HeadlessApplication не шарит PipeClient — TrayWindowTracker автономен для отправки данных. HeadlessApplication нужен PipeClient только для status polling.

PipeServer поддерживает до 5 concurrent connections (строка 73 PipeServer) — headless PipeClient (status) + TrayWindowTracker PipeClient (data) + опционально tray PipeClient (status) = 3 из 5, запас есть.

### Graceful shutdown headless

При завершении (logoff, shutdown, taskkill):
1. `SystemEvents.SessionEnding` → триггер
2. Dispose `_statusTimer`
3. `_windowTracker.Dispose()` — flush pending intervals через pipe
4. `_inputTracker.Dispose()` — unhook, flush pending events
5. `_pipeClient.Dispose()`
6. `Application.Exit()`

### TrayApplication после рефакторинга

TrayApplication становится **чистым UI-клиентом**:
- Status polling → обновление иконки и tooltip (как сейчас)
- StatusWindow — отображение статуса агента
- Меню: "Открыть Кадеро", "Статус", "Переподключиться", "Перезапустить сервис", "О программе", "Выход"
- **Нет** TrayWindowTracker, InputTracker — аналитика в headless
- **Нет** "Запускать при входе" — headless управляется инсталлятором
