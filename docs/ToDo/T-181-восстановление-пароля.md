# T-181: Восстановление доступа — сброс пароля по email

**Тип:** Task
**Приоритет:** High
**Статус:** Open
**Ветка:** `feature/email-registration` (или отдельная `feature/password-reset`)

---

## Цель

Реализовать флоу восстановления пароля через email: пользователь запрашивает сброс → получает ссылку на почту → переходит по ссылке → вводит новый пароль → логинится.

---

## Текущее состояние

- Функционал сброса пароля **отсутствует**
- Смена пароля возможна только из профиля (`POST /users/me/password`) при наличии текущего пароля
- Таблица `email_verification_tokens` (V46) уже создана для email-верификации — можно переиспользовать для reset-токенов
- `EmailService` умеет отправлять HTML-письма (`sendVerificationCode`, `sendInvitation`, `sendVerificationLink`)
- `PasswordValidator` уже реализован (мин. 8 символов, буквы + цифры)

## Backend: API

### POST `/api/v1/auth/forgot-password`

Запрос сброса пароля.

**Request:**
```json
{
  "email": "user@example.com"
}
```

**Response 200** (всегда, даже если email не найден):
```json
{
  "message": "If an account with this email exists, a password reset link has been sent"
}
```

**Логика:**
1. Нормализовать email
2. Найти пользователя по email
3. Если не найден — вернуть 200 (не раскрывать)
4. Если найден, но `auth_provider == "oauth"` и `is_password_set == false` — вернуть 200 (нельзя сбросить то, чего нет)
5. Инвалидировать все предыдущие reset-токены пользователя
6. Сгенерировать reset token (UUID, TTL 1 час)
7. Сохранить в `password_reset_tokens`
8. Отправить email со ссылкой `{FRONTEND_BASE_URL}/reset-password/{token}`
9. Аудит: `PASSWORD_RESET_REQUESTED`

**Rate limiting:**
- Per-IP: 10 запросов / 5 минут
- Per-email: cooldown 60 секунд между запросами

### POST `/api/v1/auth/reset-password`

Установка нового пароля по reset-токену.

**Request:**
```json
{
  "token": "uuid-reset-token",
  "password": "NewPass123"
}
```

**Response 200:**
```json
{
  "access_token": "jwt...",
  "token_type": "Bearer",
  "expires_in": 900,
  "user": { ... }
}
```

**Логика:**
1. Найти token в БД
2. Проверить: не использован, не истёк (1 час)
3. Валидация нового пароля (PasswordValidator)
4. Обновить `password_hash`, `is_password_set = true`
5. Пометить token как использованный
6. Инвалидировать все refresh-токены (принудительный релогин на всех устройствах)
7. Auto-login: сгенерировать JWT-токены
8. Аудит: `PASSWORD_RESET_COMPLETED`

**Ошибки:**
| Код | Code | Описание |
|-----|------|----------|
| 400 | TOKEN_NOT_FOUND | Токен не найден |
| 400 | TOKEN_EXPIRED | Токен истёк |
| 400 | TOKEN_ALREADY_USED | Токен уже использован |
| 400 | WEAK_PASSWORD | Пароль не соответствует требованиям |
| 429 | TOO_MANY_REQUESTS | Rate limit |

### POST `/api/v1/auth/forgot-password/resend`

Повторная отправка ссылки сброса.

**Request:**
```json
{
  "email": "user@example.com"
}
```

**Response 200:**
```json
{
  "message": "If an account with this email exists, a password reset link has been sent",
  "resend_available_in": 60
}
```

## Backend: БД

### Новая таблица `password_reset_tokens`

```sql
CREATE TABLE password_reset_tokens (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    token VARCHAR(255) NOT NULL UNIQUE,
    is_used BOOLEAN NOT NULL DEFAULT FALSE,
    expires_at TIMESTAMPTZ NOT NULL,
    created_ts TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_prt_token ON password_reset_tokens(token);
CREATE INDEX idx_prt_user_id ON password_reset_tokens(user_id);
```

## Безопасность

1. **Не раскрывать существование email** — ответ 200 всегда одинаковый
2. **Reset token**: UUID v4, одноразовый, TTL **1 час** (короче, чем verification — 24ч)
3. **Инвалидация сессий**: при сбросе пароля все refresh-токены пользователя отзываются (forced re-login)
4. **Rate limiting**: per-IP (10 req / 5 min), per-email cooldown (60 sec)
5. **Валидация пароля**: на бэкенде дублирует фронтенд (PasswordValidator)
6. **Аудит**: `PASSWORD_RESET_REQUESTED`, `PASSWORD_RESET_COMPLETED`, `PASSWORD_RESET_FAILED`
7. **Timing-safe**: не различать "email не найден" и "email найден" по времени ответа

## Отличия от email verification (T-180)

| Параметр | Email Verification (T-180) | Password Reset (T-181) |
|----------|---------------------------|------------------------|
| TTL токена | 24 часа | 1 час |
| Таблица | `email_verification_tokens` | `password_reset_tokens` |
| После клика | email_verified=true + auto-login | password_hash обновлён + auto-login |
| Инвалидация сессий | Нет | Да (все refresh-токены отзываются) |
| Enumeration protection | Нет (регистрация — публичная) | Да (ответ всегда одинаковый) |
| Доступен для | Новые пользователи | Существующие пользователи с паролем |
