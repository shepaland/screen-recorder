## ЭТАП 1: Системная аналитика
Используй субагента system-analyst.

1. Изучи существующий код и документацию затронутых сервисов
2. Декомпозируй задачу на технические требования
3. Спроектируй API контракты: endpoints, request/response модели, коды ошибок
4. Если нужны изменения БД — опиши схему, миграцию, индексы, влияние на партиции
5. Построй sequence diagram (Mermaid) взаимодействия между сервисами
6. Определи затронутые сервисы и зависимости
7. Зафиксируй спецификацию в docs/ (markdown)

**Критерий перехода:** спецификация готова, зависимости определены, модель данных описана.
Покажи результат и жди подтверждения перед переходом к этапу 2.

## ЭТАП 2: Разработка
Используй субагента developer.

1. Создай git-ветку: git checkout -b feature/<short-name>
2. Если нужна миграция — создай Flyway SQL (V{next}__{desc}.sql), идемпотентную и обратно-совместимую
3. Реализуй по спецификации из этапа 1: entity, repository, service, DTO, controller, валидация
4. Обеспечь: tenant isolation, structured logging (MDC), корректные HTTP-статусы, аудит
5. Напиши unit-тесты (JUnit 5 + Mockito) на бизнес-логику
6. Проверь компиляцию: ./mvnw compile
7. Прогони unit-тесты: ./mvnw test
8. Если есть фронтенд — реализуй React-компоненты, прогони npm run lint && npm run build
9. git add && git commit

**Критерий перехода:** код компилируется, unit-тесты проходят, коммит сделан.
Покажи результат и жди подтверждения перед переходом к этапу 3.

## ЭТАП 3: Деплой на dev и smoke
Используй субагента devops.

1. Собери Docker-образы затронутых сервисов
2. Задеплой в namespace dev-screen-record
3. Дождись rollout status
4. Проверь pods: kubectl -n dev-screen-record get pods
5. Проверь health: /actuator/health каждого задеплоенного сервиса
6. Если есть миграция — убедись что Flyway применил её на prg_dev

Используй субагента qa-tester.

7. Прогони smoke-тесты на dev: health endpoints, базовый CRUD, connectivity к PostgreSQL/MinIO/NATS/OpenSearch
8. Проверь что новая функциональность отвечает (минимальный happy path)

**Критерий перехода:** все поды Running, health 200, smoke passed.
Покажи результат и жди подтверждения перед переходом к этапу 4.

## ЭТАП 4: Деплой на test и тестирование
Используй субагента devops.

1. Задеплой в namespace test-screen-record
2. Дождись rollout, проверь pods и health
3. Убедись что Flyway применил миграцию на prg_test

Используй субагента qa-tester.

4. Напиши и прогони функциональные тесты по спецификации из этапа 1:
   - Happy path для каждого endpoint
   - Ошибки валидации (400)
   - Ошибки авторизации: без токена (401), чужая роль (403)
   - Tenant isolation: запрос с чужим tenant_id → 404/403
   - Граничные значения, пагинация, сортировка
5. Прогони интеграционные тесты: ./mvnw verify
6. Проверь audit_log: значимые действия зафиксированы
7. Если есть межсервисное взаимодействие — протестируй end-to-end поток

Используй субагента security-reviewer.

8. Проведи экспресс-аудит нового кода: injection, авторизация, tenant isolation, секреты в логах

**Критерий перехода:** все функциональные и интеграционные тесты passed, security-замечаний CRITICAL/HIGH нет.
Покажи полный отчёт: тест-кейсы (PASS/FAIL), security findings, покрытие.
Жди подтверждения перед переходом к этапу 5.

## ЭТАП 5: Подготовка к деплою на prod
НЕ деплой на prod. Только подготовка.

1. Сформируй release checklist:
   - Список затронутых сервисов и их Docker-образы с тегами
   - Миграции БД, которые будут применены на prg_prod
   - Изменения конфигурации (ConfigMap/Secret) если есть
   - Rollback-план: как откатить каждый компонент
2. Сформируй отчёт о готовности:
   - Результаты тестирования на test-стейджинге (количество тестов, PASS/FAIL)
   - Security audit summary
   - Известные ограничения и риски
3. Сформируй точную команду деплоя (но НЕ выполняй):
   ```
   kubectl -n prod-screen-record apply -f deploy/k8s/<service>/
   ```

Покажи всё вышеперечисленное и напиши:
**"Для деплоя на prod подтвердите командой: /project:deploy <services> prod"**
