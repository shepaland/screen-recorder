# Правила работы пайплайна команды Claude

## 1. Общий принцип

Команда работает как единый конвейер ролей:

**Бизнес-аналитик → Архитектор → Системный аналитик → UX/UI → FE/BE разработка → QA → Бизнес-аналитик**

Все роли обязаны:
- работать только в рамках своей ответственности;
- фиксировать прогресс по задаче в `docs/tasks.json`;
- не пропускать обязательные этапы;
- передавать результат следующей роли только после завершения обязательных артефактов;
- при наличии дефектов или блокеров возвращать задачу на предыдущий необходимый этап.

---

## 2. Единый источник статуса

Единый источник правды по всем задачам — файл:

`/docs/tasks.json`

В нем должна фиксироваться каждая задача и все изменения ее жизненного цикла.

### Обязательные поля задачи
Для каждой задачи должны храниться:
- `id` — порядковый номер задачи;
- `created_at` — дата и время получения;
- `title` — краткое название;
- `description` — описание исходной потребности;
- `feature_code` — код/имя фичи;
- `overall_status` — общий статус задачи;
- `role_status` — статусы по ролям;
- `artifacts` — ссылки на созданные документы и результаты;
- `history` — журнал ключевых действий.

### Возможные статусы ролей
Допустимые статусы:
- `New`
- `In Progress`
- `Waiting`
- `Done`

Если статус `Waiting`, обязательно указывать:
- какую роль ожидаем;
- что именно ожидаем.

Пример:
- `Waiting: SA`
- `Waiting: User environment for dynamic tests`
- `Waiting: BE bug fix`

---

## 3. Правила постановки задачи бизнес-аналитиком

### 3.1. Регистрация задачи
При получении новой задачи бизнес-аналитик обязан:
1. присвоить задаче следующий порядковый номер;
2. зафиксировать дату и время получения;
3. создать запись в `docs/tasks.json`;
4. инициализировать статусы всех ролей как `New`;
5. заполнить исходное описание задачи без искажения требований.

### 3.2. Старт пайплайна
После регистрации задачи бизнес-аналитик:
- уточняет бизнес-цель;
- формулирует критерии приемки;
- определяет границы задачи;
- передает задачу архитектору и/или системному аналитику в зависимости от масштаба изменений.

### 3.3. Финальная приемка
После завершения тестирования бизнес-аналитик:
- сверяет итоговый результат с исходными требованиями;
- проверяет, что все acceptance criteria закрыты;
- подтверждает завершение задачи либо возвращает ее в работу.

---

## 4. Правила работы архитектора

Архитектор подключается, если задача:
- затрагивает несколько подсистем;
- влияет на интеграции;
- требует изменения архитектурных решений;
- создает риски по масштабируемости, безопасности или сопровождению.

Архитектор обязан:
- оценить влияние изменений на систему;
- определить границы FE/BE/SA/QA работ;
- зафиксировать архитектурные ограничения и решения;
- указать обязательные нефункциональные требования;
- передать ограничения и решения системному аналитику и разработчикам.

Артефакты архитектора размещаются в:
`docs/<FEATURE>/ARCH/`

---

## 5. Правила работы системного аналитика

### 5.1. Основная задача
Системный аналитик обязан подготовить системный анализ по фиче и разместить документы в:

`docs/<FEATURE>/SA/`

### 5.2. Проверка существующих материалов
Перед началом анализа системный аналитик обязан проверить:
- существует ли уже описание связанного функционала;
- есть ли документ, покрывающий дорабатываемую область;
- есть ли зависимости от уже реализованных сценариев.

### 5.3. Версионирование
Если найден существующий документ по функционалу:
- создается новая версия документа;
- предыдущая версия переносится в архив;
- в архиве должна быть понятная версия и привязка к функционалу.

Путь архива:
`docs/<FEATURE>/SA/ARCHIVE/<FUNCTIONAL>/`

### 5.4. Обязательный результат SA
На выходе системного анализа должны быть:
- описание бизнес-сценария;
- описание системного сценария;
- ограничения;
- бизнес-правила;
- входные и выходные данные;
- изменения API/контрактов/событий/БД при необходимости;
- условия ошибок и негативные сценарии;
- декомпозиция работ для FE, BE, QA;
- критерии готовности.

После завершения SA:
- статус SA меняется на `Done`;
- статусы FE, BE, QA переводятся в `New` или `In Progress`;
- в `tasks.json` добавляется запись о декомпозиции задачи.

---

## 6. Правила работы UX/UI дизайнера

UX/UI дизайнер подключается, если задача:
- изменяет пользовательские сценарии;
- затрагивает интерфейс;
- требует прототипов, макетов, UX-flow или UI-спецификации.

Артефакты сохраняются в:
`docs/<FEATURE>/UX/`

UX/UI дизайнер обязан:
- описать пользовательский поток;
- подготовить макеты или UI-описание;
- зафиксировать состояния интерфейса;
- передать результат FE и SA.

Без UX-артефактов FE не должен интерпретировать интерфейс “по догадке”, если UI является частью задачи.

---

## 7. Правила работы разработчиков FE и BE

### 7.1. Старт разработки
Разработчики начинают работу только после того, как:
- системный анализ завершен;
- есть достаточная декомпозиция;
- понятны критерии приемки;
- отсутствуют критические блокеры по архитектуре и UX.

### 7.2. Обязательства разработчиков
FE и BE обязаны:
- реализовать только согласованный объем;
- соблюдать системный анализ и архитектурные ограничения;
- фиксировать прогресс в `tasks.json`;
- отмечать блокеры и ожидания.

### 7.3. Статические проверки
После завершения разработки разработчики обязаны выполнить статические проверки:
- линтеры;
- форматирование;
- type checks;
- unit tests;
- build checks;
- code review self-check;
- анализ очевидных регрессий.

Результаты статических проверок должны быть зафиксированы в артефактах задачи.

### 7.4. Динамическое тестирование
После статических проверок разработчики обязаны запросить у пользователя данные о среде для динамических тестов:
- URL / стенд;
- доступы;
- тестовые данные;
- особенности развертывания;
- ограничения среды.

Без явного разрешения и данных по среде динамические тесты не выполняются.

Если пользователь разрешил динамические тесты и предоставил среду:
- разработчики разворачивают функционал;
- проводят smoke / integration / manual dynamic checks;
- фиксируют результаты в `tasks.json`.

---

## 8. Правила работы QA

### 8.1. Создание тест-кейсов
QA обязан подготовить тест-кейсы по фиче и хранить их в:

`docs/<FEATURE>/QA/tests.json`

Тест-кейсы создаются после завершения системного анализа, даже если разработка еще идет.

### 8.2. Структура тест-кейсов
Каждый тест-кейс должен содержать:
- `id`
- `title`
- `preconditions`
- `steps`
- `expected_result`
- `priority`
- `blocking` — да/нет
- `status` — `Not Run`, `Pass`, `Fail`
- `defect` — если найден дефект

### 8.3. Проведение тестирования
После завершения разработки и подтверждения готовности QA:
- проходит тест-кейсы;
- проставляет `Pass/Fail`;
- фиксирует фактический результат;
- заносит дефекты в тот кейс, который их обнаружил.

---

## 9. Правила работы с дефектами

Если QA находит дефект:
- он должен быть описан внутри соответствующего тест-кейса;
- описание должно включать:
  - шаги воспроизведения;
  - фактический результат;
  - ожидаемый результат;
  - степень критичности;
  - признак блокирующего дефекта.

### 9.1. Блокирующий дефект
Если дефект блокирующий:
- QA переводит задачу разработки в `In Progress`;
- соответствующий разработчик подключается немедленно;
- дальнейшее тестирование может быть остановлено или ограничено по ситуации.

### 9.2. Неблокирующий дефект
Если дефект неблокирующий:
- QA продолжает тестирование остальных кейсов;
- все найденные дефекты фиксируются;
- по завершении цикла тестирования разработчики получают список замечаний на исправление.

### 9.3. Повторный цикл
После исправления дефектов:
- разработчики обновляют статус;
- QA повторяет тестирование;
- цикл повторяется до достижения приемлемого результата.

---

## 10. Правила передачи между ролями

Передача между ролями — это не перемещение файлов, а формальное изменение состояния задачи и фиксация результата в `tasks.json`.

Каждая передача должна содержать:
- кто передал;
- кому передал;
- когда передал;
- что именно передано;
- какие артефакты готовы;
- какие ограничения или риски остались.

---

## 11. Обязательный журнал действий

В `tasks.json` должен вестись журнал ключевых действий:
- задача создана;
- выполнен бизнес-анализ;
- выполнен системный анализ;
- сформирована декомпозиция;
- готовы UX-материалы;
- завершена FE/BE разработка;
- выполнены статические тесты;
- запрошена среда для динамических тестов;
- выполнены динамические тесты;
- созданы тест-кейсы;
- проведено тестирование;
- найден дефект;
- дефект исправлен;
- выполнен ретест;
- задача принята бизнес-аналитиком.

---

## 12. Правила оркестрации

Связывающая команда или оркестратор обязаны:
- запускать роли строго по порядку жизненного цикла;
- не позволять перескакивать обязательные этапы;
- следить за актуальностью `docs/tasks.json`;
- возвращать задачу на предыдущий этап при обнаружении дефектов или неполноты артефактов;
- проверять наличие обязательных файлов перед передачей на следующий этап.

Оркестратор не заменяет роль, а управляет последовательностью и контролем переходов.

---

## 13. Правила готовности этапов

### Ready for Development
Задача считается готовой к разработке, если:
- зарегистрирована бизнес-аналитиком;
- завершен системный анализ;
- есть архитектурные решения, если требуются;
- есть UX-материалы, если затрагивается UI;
- есть декомпозиция FE/BE/QA.

### Ready for QA
Задача считается готовой к тестированию, если:
- разработка завершена;
- статические проверки пройдены;
- результаты зафиксированы;
- тестируемый инкремент доступен;
- QA имеет тест-кейсы.

### Ready for Business Acceptance
Задача считается готовой к финальной приемке, если:
- QA завершил цикл тестирования;
- все блокирующие дефекты устранены;
- результаты тестирования зафиксированы;
- финальный статус разработки и тестирования обновлен.

---

## 14. Ограничения

Ролям запрещено:
- пропускать обновление `docs/tasks.json`;
- начинать кодирование без результатов SA;
- закрывать задачу без QA;
- менять требования без фиксации у BA;
- считать задачу завершенной без финальной проверки бизнес-аналитика.

---

## 15. Предпочтительный жизненный цикл задачи

1. BA регистрирует задачу в `docs/tasks.json`
2. Архитектор оценивает влияние, если нужно
3. SA анализирует задачу и готовит документы
4. UX/UI готовит материалы, если нужно
5. SA декомпозирует работу на FE/BE/QA
6. FE/BE реализуют решение
7. FE/BE проводят статические проверки
8. FE/BE запрашивают у пользователя среду для динамического тестирования
9. При наличии среды FE/BE проводят динамические тесты
10. QA выполняет тест-кейсы
11. При дефектах задача возвращается разработчикам
12. После исправления QA выполняет ретест
13. BA проводит финальную проверку соответствия исходным требованиям
14. Задача переводится в `Done`

---

## 16. Критерий завершения задачи

Задача может быть переведена в `Done` только если:
- все обязательные роли завершили свою часть работы;
- все артефакты созданы и сохранены в нужных директориях;
- в `docs/tasks.json` отражена полная история обработки;
- QA завершил тестирование;
- бизнес-аналитик подтвердил соответствие исходным требованиям.
